** Universal Editor Contract
:PROPERTIES:
:CUSTOM_ID: universal-editor-contract
:END:
Clean Emacs must behave like a useful editor for standalone files, not only for
known projects. Opening a throwaway file in =~/Documents=, =/tmp=, or a study
folder should provide the best local behavior available without requiring a Git
repo, flake, Cargo workspace, or project file.

- Universal baseline: major mode, syntax highlighting, indentation, =M-/=
  completion, snippets where tracked, file/buffer/keyword completion fallbacks,
  navigation keys, and explicit formatting where a formatter exists.
- Project intelligence: LSP diagnostics, xref, project-aware completion, NixOS
  option graphs, Rust workspaces, and other deep language features are additive
  when project metadata exists.
- Degraded standalone behavior should stay quiet and useful. A missing project
  file should not turn ordinary editing into a broken surface.

Current baseline examples:

- Standalone =.nix= opens in =nix-ts-mode= with Tree-sitter, =nixd=, =nixfmt=,
  and fallback completion for local bindings and keywords.
- Standalone =.lua=, =.py=, =.sh=, =.json=, and =.yaml= open in TS modes with
  completion fallbacks and explicit formatters.
- Standalone =.qml= uses classic =qml-mode=, QML snippets, =qmlls-wrapped=, and
  =qmlformat= because no better built-in QML TS mode is currently active.
- Standalone =.rs= uses =rust-ts-mode= and =rustfmt=, but waits for
  =Cargo.toml= or =rust-project.json= before starting rust-analyzer so study
  files do not emit workspace errors.
- Standalone =.c= uses =c-ts-mode= and =clang-format=; =clangd= can add
  project intelligence when it has enough context.

** Rich Language Contract
:PROPERTIES:
:CUSTOM_ID: rich-language-contract
:END:
The universal editor baseline is not the final target. Clean Emacs should aim
for rich language behavior across every configured language, not only Nix and
not only files opened inside known repos.

For each configured language, define and audit the full language surface:

- Server choice: use the best practical language server, and prefer parity with
  the already-working NVF setup when that setup behaves better.
- Standalone behavior: opening a study/playground file outside a repo should
  still provide rich completion, hover/docs, diagnostics, snippets, syntax,
  indentation, and formatting wherever the tooling can honestly support it.
- Project behavior: project metadata such as =flake.nix=, =Cargo.toml=,
  =pyproject.toml=, =compile_commands.json=, or QML import metadata should add
  deeper intelligence, not be a prerequisite for a useful editor.
- Completion behavior: LSP completion should be combined with snippets,
  keywords, paths, buffer words, and local symbols so a weak language-server
  context does not make =M-/= feel dead.
- Diagnostics behavior: standalone files should not spam project-metadata errors
  unless the warning is genuinely useful for the edit at hand.
- Formatting behavior: =SPC c f= should be explicit and available for every
  configured language with a known formatter.

Current Nix parity decision:

- NVF uses =nixd= for Nix. Clean Emacs was originally using =nil=, which made
  standalone files noticeably less rich.
- A direct clean-Emacs comparison on the same standalone Nix file showed =nixd=
  completing NixOS/service options and =lib.mk*= helpers where =nil= did not.
- Clean Emacs now uses =nixd= for Nix to match the richer standalone behavior
  and align with NVF's server choice.

Target language audit set:

- Nix: =nixd= standalone option completion, flake/NixOS/Home Manager
  context.
- Lua: =lua-language-server= settings, runtime/library awareness, standalone
  and config-file completion.
- QML/Quickshell: =qmlls= behavior, import paths, Quickshell metadata, and
  snippet completion.
- Python: =basedpyright= behavior, standalone files, project files, and
  import/completion quality.
- Rust: =rust-analyzer= behavior for study files versus Cargo projects.
- C/C++: =clangd= behavior for standalone files versus compile database
  projects.
- Shell: =bash-language-server= behavior and =shfmt=.
- JSON/YAML: language-server behavior, schemas where useful, and Prettier
  formatting.
- TypeScript/JavaScript/CSS/HTML: language-server parity with NVF and formatter
  availability.
- Org/Markdown/Text: Harper diagnostics and prose completion behavior.

*** 2026-06-07 parity audit findings
:PROPERTIES:
:CUSTOM_ID: parity-audit-2026-06-07
:END:
First read-only pass compared static NVF settings, clean Emacs settings, and
headless standalone-file runtime behavior. Do not treat this as a final tuning
decision; it is the map for the next slices.

- Nix was the clearest mismatch:
  - NVF uses =nixd=.
  - Clean Emacs used =nil=.
  - A direct clean-Emacs comparison showed =nixd= gives richer standalone Nix
    completion for =services.= and =lib.mk*=.
  - Clean Emacs now adds =nixd= to the Emacs runtime tools and routes Nix Eglot
    buffers to =nixd=.
- QML is mostly aligned in server choice:
  - NVF uses =qmlls= through =qmlls-wrapped= with Qt/Quickshell import paths.
  - Clean Emacs uses =qmlls-wrapped=.
  - Clean Emacs still uses classic =qml-mode=; no better built-in QML TS mode is
    currently active.
- Lua, Python, Rust, C/C++, shell, JSON, YAML, TypeScript, HTML, and CSS need
  per-language runtime checks before changes:
  - Static configuration suggests many servers are similar or equivalent.
  - Headless NVF standalone probing did not start every expected code server,
    so do not blindly assume NVF runtime behavior from static configuration.
  - Clean Emacs currently attaches more servers in the headless standalone probe,
    except Rust, which intentionally waits for =Cargo.toml= or
    =rust-project.json= to avoid standalone workspace errors.
- Formatter parity is not complete:
  - Clean Emacs has explicit formatters for Nix, Lua, QML, Rust, C/C++, Python,
    shell, JSON, and YAML.
  - Clean Emacs does not yet have explicit TypeScript/JavaScript/CSS/HTML
    formatter entries, even though those languages have LSP coverage.

Next action order:

1. Rebuild/restart and validate =nixd= in clean Emacs on standalone and project
   Nix files.
2. Audit each remaining language in focused slices, using the same standard:
   rich standalone behavior first, project metadata as additive depth.
3. Add missing explicit formatter entries where the language tooling already
   exists.

** Daily Notes Flow
:PROPERTIES:
:CUSTOM_ID: daily-notes-flow
:END:
- =SPC n i= opens the inbox for unsorted thoughts.
- =SPC n d= opens today's daily note.
- =SPC n c= captures a note, task, daily entry, topic entry, or project entry.
- =SPC n f= finds a note under =~/Documents/notes=.
- =SPC n s= searches all notes under =~/Documents/notes=.
- =SPC n o= opens the notes root in Dired.
- =SPC n a= opens the agenda after refreshing note discovery.
- Agenda custom views include =d= daily review, =w= week, =i= inbox, and =f= follow-up search.
- =SPC n T= opens the TODO agenda after refreshing note discovery.
- =SPC n r= opens today's daily note beside the daily review agenda.
- =SPC n R= refreshes agenda file discovery manually.
