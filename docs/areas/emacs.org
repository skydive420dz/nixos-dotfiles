** Universal Editor Contract
:PROPERTIES:
:CUSTOM_ID: universal-editor-contract
:END:
Clean Emacs must behave like a useful editor for standalone files, not only for
known projects. Opening a throwaway file in =~/Documents=, =/tmp=, or a study
folder should provide the best local behavior available without requiring a Git
repo, flake, Cargo workspace, or project file.

- Universal baseline: major mode, syntax highlighting, indentation, =M-/=
  completion, snippets where tracked, file/buffer/keyword completion fallbacks,
  navigation keys, and explicit formatting where a formatter exists.
- Project intelligence: LSP diagnostics, xref, project-aware completion, NixOS
  option graphs, Rust workspaces, and other deep language features are additive
  when project metadata exists.
- Degraded standalone behavior should stay quiet and useful. A missing project
  file should not turn ordinary editing into a broken surface.

Current baseline examples:

- Standalone =.nix= opens in =nix-ts-mode= with Tree-sitter, =nixd=, =nixfmt=,
  and fallback completion for local bindings and keywords.
- Standalone =.lua=, =.py=, =.sh=, =.json=, and =.yaml= open in TS modes with
  completion fallbacks and explicit formatters.
- Standalone =.qml= uses =sk-qml-ts-mode= when the =qmljs= Tree-sitter parser is
  available, with classic =qml-mode= as fallback. QML snippets,
  =qmlls-wrapped=, and =qmlformat= remain active.
- Standalone =.rs= uses =rust-ts-mode= and =rustfmt=, but waits for
  =Cargo.toml= or =rust-project.json= before starting rust-analyzer so study
  files do not emit workspace errors.
- Standalone =.c= uses =c-ts-mode= and =clang-format=; =clangd= can add
  project intelligence when it has enough context.

** Rich Language Contract
:PROPERTIES:
:CUSTOM_ID: rich-language-contract
:END:
The universal editor baseline is not the final target. Clean Emacs should aim
for rich language behavior across every configured language, not only Nix and
not only files opened inside known repos.

For each configured language, define and audit the full language surface:

- Server choice: use the best practical language server, and prefer parity with
  the already-working NVF setup when that setup behaves better.
- Standalone behavior: opening a study/playground file outside a repo should
  still provide rich completion, hover/docs, diagnostics, snippets, syntax,
  indentation, and formatting wherever the tooling can honestly support it.
- Project behavior: project metadata such as =flake.nix=, =Cargo.toml=,
  =pyproject.toml=, =compile_commands.json=, or QML import metadata should add

  keywords, paths, buffer words, and local symbols so a weak language-server
  context does not make =M-/= feel dead.
- Diagnostics behavior: standalone files should not spam project-metadata errors
  unless the warning is genuinely useful for the edit at hand.
- Formatting behavior: =SPC c f= should be explicit and available for every
  configured language with a known formatter.

** Powerhouse Editor Experiment Contract
:PROPERTIES:
:CUSTOM_ID: powerhouse-editor-experiment-contract
:END:
This branch experiments with a richer IDE/LSP-oriented clean Emacs while keeping
=main= as the known-good functional checkpoint. The experiment must improve code
intelligence without turning the editor into a pile of unrelated patches.

Scope:

- Primary focus: LSP-powered completion and Tree-sitter highlighting.
- Secondary focus: diagnostics, hover/docs, xref/navigation, and formatting only
  after the LSP/completion/highlighting owner model is clear.
- Out of scope unless explicitly agreed: unrelated keymap redesign, UI churn,
  terminal rewrites, notes workflow changes, Magit changes, dashboard changes,
  shell changes, and unrelated package experiments.

Working rules:

- Do not change general key functionality unless the change is proposed and
  agreed on first.
- Do not manipulate keymaps as a side effect of testing LSP/completion unless
  the key behavior is part of the current slice.
- Do not patch incrementally around symptoms until the ownership model is known.
- Identify candidate improvements and present them as options before making
  assumptions.
- Prefer one coherent owner for each capability over many small overlapping
  packages.
- Each slice must have a stated success condition and rollback point.
- If a tool is removed, replaced, or made secondary, document what owns that
  responsibility now.
- No half-implemented helper code should stay active. Experimental code either
  works, is disabled with a clear note, or is reverted.

Operational definition of surface:

- Surface is every place behavior can be configured, fail, drift, or require
  ownership.
- Fewer packages is not automatically less surface.
- Many small packages glued together can be more surface than one larger mature
  package if they add boundaries, assumptions, config files, failure points, or
  unclear ownership.

Target language set for this experiment:

- Nix
- Lua
- QML
- Python
- Shell
- JSON
- Emacs Lisp
- Org
- Rust
- C

For each target language, track:

- major mode and Tree-sitter mode
- parser source and install path
- LSP server
- LSP client owner
- completion owner and sources
- snippet behavior
- diagnostics owner
- hover/docs behavior
- formatter
- standalone-file behavior
- project-file behavior
- known limitations

Missing items to keep explicit:

- Diagnostics owner: decide whether Flymake, Flycheck, or LSP UI owns error
  display and navigation.
- Completion UI owner: decide whether the richer stack uses Corfu, Company, or
  a hybrid only if the boundary is defensible.
- Formatting policy: keep manual formatting unless format-on-save is explicitly
  agreed per language.
- Performance budget: measure startup/reload, first LSP attach, idle CPU, and
  obvious UI lag after major changes.
- Test matrix: every slice should test at least one project file and one
  standalone file outside a repo.

Initial slice proposal:

1. Pick the IDE owner for one language only.
2. Prove completion, docs, diagnostics, xref, syntax highlighting, and formatting
   on that language.
3. Document the result.
4. Only then scale the pattern to the next language.

*** Slice 2 Lua lsp-mode/company proof
:PROPERTIES:
:CUSTOM_ID: powerhouse-slice-2-lua-lsp-mode-company

Date: 2026-06-09

Decision:

- Lua is the first proof language because it is familiar enough to evaluate
  completion/doc behavior quickly.
- The proof path is =lua-ts-mode= / =lua-mode= + =lua-language-server= +
  =lsp-mode= + =company=.
- Other languages keep the existing Eglot/Corfu path during this slice.

Implementation boundary:

- =sk-lsp.el= remains the Eglot owner.
- =sk-lsp-mode.el= owns only lsp-mode proof slices.
- =sk/lsp-mode-owned-modes= lets Eglot skip modes intentionally handled by
  =lsp-mode=.
- Corfu remains global default, but Lua proof buffers disable local =corfu-mode=
  and enable =company-mode= so there is one in-buffer completion UI.
- Global completion refresh respects =sk/lsp-mode-owned-modes= so reloads do
  not re-enable Corfu/Cape fallback ownership inside Lua proof buffers.
- Lua proof buffers use a strict CAPF list: =lsp-completion-at-point= only.
  Generic fallback sources such as =sk/yas-capf=, =cape-file=, =cape-dabbrev=,
  and =cape-keyword= are intentionally removed from Lua buffers during this
  proof so old-stack candidates do not contaminate the lsp-mode/company test.
- General keymaps were not changed.

LuaLS settings used by the proof:

- =lsp-clients-lua-language-server-bin= and
  =lsp-clients-lua-language-server-command= point at the Nix-provided
  =lua-language-server=.
- =lsp-clients-lua-language-server-main-location= points at the Nix package's
  =share/lua-language-server/main.lua= when present.
- =lsp-lua-runtime-version= is =LuaJIT= to match the Neovim Lua config use case.
- =lsp-lua-diagnostics-globals= includes =vim= and =hl=.
- =lsp-lua-workspace-library= reuses the existing Neovim runtime metadata path.
- Lua call and keyword snippets are enabled for the proof.
- Diagnostics use =flycheck= through =lsp-diagnostics-provider :flycheck=.
  This avoids the older Flymake path and the extra =lua-ts-flymake-luacheck=
  backend.
- Existing code helpers dispatch by active backend: =lsp-mode= first, then
  Eglot. The keymap stays stable.
- Breadcrumbs use =lsp-headerline-breadcrumb-mode= with icons enabled and the
  concise segment set =(project file symbols)=. Sky theme owns the breadcrumb
  faces.
- Breadcrumb icons use the upstream lsp-mode provider path: =lsp-treemacs= for
  file/symbol icons and =all-the-icons= for the separator/icon font side.
  System font coverage includes =emacs-all-the-icons-fonts= via Nix.
- Breadcrumb provider loading is explicit in the Lua proof hook so lsp-mode does
  not render text-only fallbacks before =lsp-treemacs= and =all-the-icons= are
  available.
- Breadcrumb PNG icons stay on the upstream =lsp-treemacs= / =lsp-icons= path.
  PGTK headerline rendering did not display those PNGs while the image display
  spec carried =:mask heuristic= and =:background unspecified=, so
  =sk-lsp-mode.el= sanitizes that metadata after =lsp-icons= builds the icon.
  This keeps real PNG icons instead of falling back to Nerd Font glyphs.
- Company popup navigation mirrors the existing Corfu navigation contract:
  =C-j= next, =C-k= previous, =C-l= accept, =C-h= abort. The shared
  completion wrappers dispatch to Company when Company is active and to Corfu
  when Corfu is active.
- Company documentation is explicit, not automatic, to avoid stacked popup
  surfaces: =M-d= opens the Company documentation buffer while the completion
  menu is active; =SPC c k= remains the code-hover/doc command.
- Company uses one popup surface for the Lua proof:
  =company-frontends=(company-pseudo-tooltip-frontend)=. The default echo
  metadata and single-candidate preview frontends are disabled to avoid stacked
  or changing popup surfaces while typing.

** Daily Notes Flow
:PROPERTIES:
:CUSTOM_ID: daily-notes-flow
:END:
- =SPC n i= opens the inbox for unsorted thoughts.
- =SPC n d= opens today's daily note.
- =SPC n c= captures a note, task, daily entry, topic entry, or project entry.
- =SPC n f= finds a note under =~/Documents/notes=.
- =SPC n s= searches all notes under =~/Documents/notes=.
- =SPC n o= opens the notes root in Dired.
- =SPC n a= opens the agenda after refreshing note discovery.
- Agenda custom views include =d= daily review, =w= week, =i= inbox, and =f= follow-up search.
- =SPC n T= opens the TODO agenda after refreshing note discovery.
- =SPC n r= opens today's daily note beside the daily review agenda.
- =SPC n R= refreshes agenda file discovery manually.

The intended rhythm is small and repeatable:

- Capture loose thoughts to the inbox when they do not yet have a home.
- Use the daily note as the working surface for the day.
- During review, process inbox items into tasks, project notes, topic notes, or archive.
- Clarify TODOs before carrying them forward.
- Mark follow-ups and choose tomorrow's first focus.

New daily files receive this review checklist automatically. Existing personal
notes are left untouched.
