* Pinned Nix 2.34.8 Behavior

Both =nix store optimise= and =nix-store --optimise= call the same full-store
optimizer.  The pinned implementation:

1. snapshots database-valid store paths;
2. recursively examines regular files and, on this Linux build, symlinks;
3. skips owner-writable regular files, directories, and special objects;
4. keys regular files by content plus executable/non-executable status;
5. keys symlinks by target bytes;
6. creates =/nix/store/.links/<NAR hash>= for the first occurrence;
7. atomically replaces later identical objects with hard links.

There is no optimizer dry-run.  =--readonly-mode= is not a dry-run.  A real
invocation starts populating =.links= immediately and is nontransactional
across the whole store, although each individual replacement is atomic.

The full optimizer has no exclusive whole-store optimizer lock.  It coordinates
each valid path with GC through temporary roots, but the pinned source still
documents a race with GC's =.links= cleanup.  A real run therefore must not
overlap GC, another optimizer, or a changing-store measurement window.

The first run with an empty =.links= pool hashes every eligible object and
creates an index entry even for content that is unique.  Successful service
exit can still include partial-skip warnings such as link-count or filesystem
limit failures, so the journal and independent before/after values matter.

** Exact External Equivalence Scan

Because Nix has no dry-run, a separate low-priority Python process estimated
the same equality relation without invoking the Nix client or changing links.
It:

1. queried =ValidPaths= through SQLite's read-only WAL view;
2. required that the valid-path list exactly equal the store's non-=.links=
   top-level list;
3. required =.links= to be empty and all candidate link counts to be one;
4. skipped the =21,962= regular files unique by size and executable status;
5. SHA-256 hashed =1,104,769= regular files totaling =40,370,028,054= bytes;
6. grouped regular files by size, executable status, and digest;
7. grouped symlinks by exact target bytes;
8. repeated the path/database/bookkeeping checks at completion.

SHA-256 grouping has the same practical equivalence relation as Nix's
per-object NAR hash when object type and executable status are separated.
Cryptographic collisions were not followed by a byte-for-byte third pass.

** Slice O2 - Runtime Verification

After the service reaches =inactive (dead)= following successful completion:

1. require =Result=success= and =ExecMainStatus=0=;
2. inspect the complete service journal for warnings and partial skips;
3. measure duration and the optimizer's reported files/bytes linked;
4. make one allocated-store and filesystem before/after comparison;

6. compare filesystem inode use and temp-root/reserve state;
7. prove current/profile/booted outputs, generations, GC timer, UKIs, and
   legacy entries are unchanged;

Do not run GC, a build, activation, reboot, or boot cleanup as part of this
verification.

** Slice O3 - Recurrence Policy

Only after O2, choose one:

- keep the result as one-shot-only maintenance;
- add an MSI-only periodic =nix.optimise= timer in a new source candidate;
- defer future optimization until store growth makes another measured run
  worthwhile.

If a timer is selected:

- keep =auto-optimise-store=false= initially;
- use =Persistent=false= to avoid activation catch-up;
- schedule it on a different day or with a generous gap from GC;
- keep boot retention and cleanup out of the candidate;
- evaluate, pure-build, inspect, accept, activate, and runtime-verify as
