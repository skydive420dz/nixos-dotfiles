* Purpose And Verdict
:PROPERTIES:

This is the focused companion to [[file:2026-07-14-nixos-system.org][the 2026-07-14 NixOS system audit]].  It
maps the complete active Quickshell configuration, measures the running shell,
separates confirmed defects from performance hypotheses, and defines small
implementation slices with measurable acceptance gates.

The audit began read-only.  It did not edit QML, open a picker, change media
playback, restart Quickshell, rebuild or switch NixOS, change the system profile,
or reboot the machine.  The only writes made after the evidence pass are this
documentation and its index links.

Evidence labels used below:

- =CONFIG=: declared by the repository or installed QML type metadata.
- =ACTIVE=: observed on the running machine.
- =HISTORY=: journal, coredump, or completed-unit accounting from an older run.
- =VALIDATION=: read-only parser, linter, or syntax-check result.
- =INFERENCE=: a likely cost or cause that still requires an A/B test.

** Executive verdict
:PROPERTIES:
:CUSTOM_ID: executive-verdict
:END:
The current shell is stable and thoughtfully modular, but it is not yet lean.
The highest-value work is architectural rather than cosmetic:

1. Make the dynamic bar a small render surface instead of sharing a blurred,
   full-output window with the static screen frame.
2. Replace per-bar shell polling with one shared, mostly event-driven state
   backend using the native Quickshell services already installed.
3. Share or reduce the 24 fps media analyzer and measure it independently.
4. Virtualize and pre-thumbnail the 514 MiB live wallpaper collection.
5. Put safety and validation ahead of those changes: fix shell quoting, the CAVA
   temporary-file leak, the three layout defects, and the live-link/restart
   failure boundary first.

The live media-playing baseline is about =7.7-8.1%= of one CPU core, roughly
=177-185 MiB= of cgroup memory, about =205 MiB= process PSS, and =92 MiB= of
NVIDIA framebuffer allocation.  Those figures are not an idle baseline: Brave
was playing media and one CAVA analyzer was active.  No playback state was
changed merely to manufacture a second result.

** QS-01: wallpaper shell boundary
:PROPERTIES:
:CUSTOM_ID: qs-01-wallpaper-shell-boundary
:END:
[[../../config/quickshell/modules/wallpaper/WallpaperStore.qml][WallpaperStore.qml]] uses =JSON.stringify= as if it were a shell escaping
function in the video thumbnail command, state save/load, and directory list.
JSON produces double-quoted strings; Bash still expands =$variables=, backticks,
and =$(commands)= inside double quotes.  A downloaded wallpaper with a crafted
filename can execute code as the logged-in user when its thumbnail runs or state
is saved.  The current 23 filenames inspected are benign, so no current trigger
was observed.

Use argv-based processes or pass paths as positional parameters to a fixed helper.
Write state and thumbnails to a temporary file followed by an atomic rename.

** QS-02: CAVA temporary files
:PROPERTIES:
:CUSTOM_ID: qs-02-cava-temporary-files
:END:
[[../../config/quickshell/modules/media/MediaAnalyzer.qml][MediaAnalyzer.qml]] creates a temporary config, installs an EXIT trap, and then
runs =exec cava=.  A successful =exec= replaces Bash, so Bash never executes its
trap.  The active machine has 254 matching mode-0600 configs in =/tmp=, 190 bytes
each; four were created during the current service run.

Use one declarative/store-backed CAVA config or a stable file under the private
runtime directory.  This is a small, behavior-preserving first fix.

** QS-03 and QS-04: render topology and analyzer
:PROPERTIES:
:CUSTOM_ID: qs-03-and-qs-04-render-topology-and-analyzer
:END:
[[../../config/quickshell/root/Bar.qml][Bar.qml]] sets its implicit height to the output height, and
[[../../config/quickshell/common/BarFrame.qml][BarFrame.qml]] paints/clears that full Canvas.  Hyprland then applies blur to
the =qs-bar= namespace.  The analyzer updates inside the same QQuickWindow up to
24 times per second.  Geometry and update rate are definite; how much damage and
blur work propagates is not yet proven.

First share one analyzer backend and reduce or make its cadence configurable.
Then A/B the existing unified surface against a 34 px dynamic bar plus a separate
static, input-transparent frame.  Preserve the approved unified visual and mask
invariants; do not blindly remove the frame.

The installed =PwNodePeakMonitor= is an event-driven alternative if aggregate
level animation is visually acceptable.  If CAVA stays, run it once, add sample
freshness/restart handling, lower the frame rate to 12-15 fps, and expose the
same levels to every bar.  Today a stalled CAVA can leave old levels nonempty and
prevent the fallback from resuming.

** QS-05: per-output backend duplication
:PROPERTIES:
:CUSTOM_ID: qs-05-per-output-backend-duplication
:END:
Every connected output adds a complete =Bar=, including five poll loops,
=Workspaces=, =WindowTitle=, =Tray=, =Clock=, and =MediaAnalyzer=.  With HDMI
connected, the current code doubles status process churn to about 28,560 process
executions/hour, opens two Pulse/CAVA captures during playback, and gives both
status instances permission to emit the same battery OSD transition.

Move stateful backends to =ShellRoot= or true singletons and keep only cheap view
delegates per output.  As a low-risk first step, centralize the existing pollers
unchanged; replace each source with a native model in later commits.

** QS-06: wallpaper lifecycle and picker
:PROPERTIES:
:CUSTOM_ID: qs-06-wallpaper-lifecycle-and-picker
:END:
[[../../config/quickshell/modules/wallpaper/WallpaperSelector.qml][WallpaperSelector.qml]] uses =Flickable + Grid + Repeater=.  Opening it creates
all 23 cards.  Eighteen non-video cards load original sources without
=sourceSize= and with =cache: false=.  Their one-frame raw RGBA dimensions total
about 359.7 MiB before textures; animated frames can cost more.  Five video cards
each start a Bash freshness check and can launch five FFmpeg decoders concurrently
when thumbnails are missing or stale.

The store also refreshes the list immediately before opening, assigns a new
array even when unchanged, and can rebuild the delegates twice.  Startup begins
with =wallpaper-003.gif= while the persisted state is loaded asynchronously, so
every restart can briefly create GIF decoders before switching to the current
PNG.  Static wallpapers disable the shared image cache and enable mipmaps per
output.  Animated/video outputs never pause for battery state or DPMS.

Use a virtualized =GridView=, display-sized thumbnails for every format, one
bounded thumbnail worker, content/mtime cache keys, and cache pruning.  Load
state before constructing a decoder.  Then add an explicit animated-wallpaper
power policy and convert the 143.9 MB GIF to MP4/WebM for an A/B test.

** QS-07: production versus development mode
:PROPERTIES:
:CUSTOM_ID: qs-07-production-versus-development-mode
:END:
A Nix build verifies the user-unit and package closure but not the bytes under the
out-of-store QML link.  Historical development evidence shows why the distinction
matters: on 2026-06-09 invalid configurations exited 255, restarted at the
effective 100 ms =RestartUSec=, and hit the five-start limit twice.  Forty-one
older May Quickshell coredump records are retained.  None of these failures is
current: the last 30 days and current run have no crash, OOM, or restart loop.

Provide an explicit stable/store-backed mode and a separate opt-in live-link
development mode.  If live mode remains the default, require lint before manual
restart, set a one- or two-second restart delay, and make failure state obvious.

** QS-08 through QS-10: correctness and races
:PROPERTIES:
:CUSTOM_ID: qs-08-through-qs-10-correctness-and-races
:END:
- Each =Workspaces= reads global =Hyprland.focusedMonitor=, so two bars show the
  focused output's active workspace rather than their own; each also repeats the
  same =hyprctl workspaces -j= query.
- =WindowTitle= clears the active title on every =closewindow= event, including
  an unrelated background window.
- Network interface changes do not reset prior byte counters/timestamps, so a
  handoff can graph a false spike.  The first connected =nmcli= row is selected,
  not necessarily the default-route interface, and identity can lag 15 seconds.
- Dual status instances race duplicate battery OSD writes.
- Launcher, clipboard, and OSD write target/payload and signal as separate files.
  A FileView event that arrives while the reader =Process= is already running can
  coalesce.  Repeating volume/brightness binds make this a real hot path.
- Launcher and clipboard do not stop a pending 150 ms close timer when reopened;
  the old timer can clear a newly typed query.
- Clipboard delete clamps against =results.length - 2= after changing the model
  and can jump an extra row.
- The linter-confirmed layout defects leave three child heights undefined under
  =ColumnLayout= ownership.

Native Hyprland workspaces/active-toplevel state fixes the first two without
shell queries.  Typed Quickshell IPC can carry overlay target and OSD payload
atomically.  Keeping the external hardware scripts still lets volume keys work
when Quickshell is down; only the UI notification transport needs to become IPC.

** QS-11: duplicate theme lifecycle
:PROPERTIES:
:CUSTOM_ID: qs-11-duplicate-theme-lifecycle
:END:
Both Quickshell and Mako run =theme-select apply= as concurrent =ExecStartPre=
jobs.  Each invocation regenerates every consumer theme, repeatedly forks =jq=,
writes the same mutable files, and updates the same user-manager environment.
For the current Quickshell start, the journal span was about 0.58 seconds from
=Starting= to its main process and about 0.86 seconds to =Configuration Loaded=.
Systemd's monotonic unit accounting put both Mako and Quickshell at about 1.11
seconds from inactive transition to main-process start.  The exact duration is
small; the duplicate work and concurrent write ownership are the real defects.

=Theme.qml= already watches =quickshell-signal= and reloads in place.  Despite
that, =theme-select toggle= also restarts Quickshell; the restart then regenerates
the global theme a second time and recreates pollers/CAVA.

Create one ordered =theme-select-apply.service= or generate the initial theme at
Home Manager activation.  Make Quickshell and Mako depend on that result, and
remove the Quickshell restart from normal theme toggles.

** QS-12 and QS-13: clipboard and launcher
:PROPERTIES:
:CUSTOM_ID: qs-12-and-qs-13-clipboard-and-launcher
:END:
Clipboard preview conversion has no time/memory limit, cannot cancel a stale
ImageMagick job, never deletes preview PNGs during a session, and keys them only
by Cliphist ID.  After a wipe/ID reuse it can show stale content.  A failed
conversion publishes a =file://= URL without checking output existence or exit
status, silently producing a broken preview.  Entries remain in QML memory after
close.  The broader Cliphist service persists about 750 entries and already
documents that 1Password Linux may not mark secrets.

Use content-addressed runtime previews, atomic writes, success-aware publication,
bounded ImageMagick resources, cancellation/coalescing, and cleanup.  Preserve
the existing positional-argument handling for raw clipboard data.

Launcher discovery reparses desktop files and returns JSON on every open even
when its in-memory app list is populated.  The measured 60-74 ms is acceptable
today but unnecessary.  The helper also incompletely implements desktop-entry
masking, =OnlyShowIn=, =TryExec=, working directories, and D-Bus activation;
malformed =Exec= parsing falls back to raw shell text.  Adopt the installed
=DesktopEntries.applications= model or cache a spec-correct helper.  Preserve
UWSM scoping explicitly when launching.

** QS-14 and QS-15: power and hygiene
:PROPERTIES:
:CUSTOM_ID: qs-14-and-qs-15-power-and-hygiene
:END:
- The clock wakes every second for =HH:mm= and date, and runs once both from
  =Component.onCompleted= and =triggeredOnStart=.  Align one shared timer to the
  next minute, then use a 60-second interval.
- Numeric network rates and raw byte properties are still calculated/passed even
  though the current UI renders only graph samples.
- Most process parse/exit errors are swallowed and blocking CLIs have no timeout;
  a feature can silently freeze while the unit remains healthy.
- =WallpaperStore.configHome= is unused; OSD =kind= is emitted but unread.
- The user PATH points to =ffmpeg-full= while the service separately retains the
  standard =ffmpeg= output for =SKY_FFMPEG=.
- The Quickshell flake input is unused and misleading.
- The service description says "Quickshell bar" although it owns the wallpaper,
  launcher, clipboard, OSD, tray, and status system.
- =QT_DISABLE_HW_TEXTURES_CONVERSION=1= disables GPU texture conversion for Qt
  video.  Qt documents that GPU conversion can significantly reduce CPU, but the
  variable may be a valid rendering workaround on this AMD/NVIDIA hybrid host.
  Treat removal as a controlled video-wallpaper A/B, not a blind cleanup.  See
  [[https://doc.qt.io/qt-6/advanced-ffmpeg-configuration.html][Qt 6.11 Advanced FFmpeg Configuration]].

* Slice-Based Implementation Roadmap
:PROPERTIES:

without bundling unrelated visual changes.

:PROPERTIES:
:CUSTOM_ID: slice-0-reproducible-checks-and-measurements

  Python parsing against the actual source directory.

start or restart a shell; known false positives are named rather than ignored

* Performance Acceptance Matrix
:PROPERTIES:
:CUSTOM_ID: performance-acceptance-matrix
:END:
Use the same output topology, wallpaper, media state, sampling interval, and
power profile before comparing a slice.

| Workload | Primary target | Secondary guardrail |
|----------+----------------+---------------------|
| One-output, no media | establish first; target <=1% one core | no steady helper churn |
| One-output, media playing | <=4% one core | one analyzer, lower GPU burst duty |
| Two-output, media playing | close to one-output backend cost | correct per-output workspace/state |
| Picker cold open | one bounded thumbnail worker | no multi-FFmpeg fan-out |
| Picker warm open/close | small peak; return near pre-open current memory | no stale or blank thumbnails |
| Eight-hour normal use | cgroup peak <=250 MiB excluding deliberate conversion jobs | stable FD/temp-file counts |
| Theme toggle | no Quickshell restart | one generator invocation |
| Rapid OSD keys | latest value wins, timer extends | external hardware action still works without QS |

The exact idle target is provisional because idle playback was not forced during
this audit.  Capture it before using the table as a pass/fail gate.

* Build, Activation, And Hand-Test Boundary
:PROPERTIES:
:CUSTOM_ID: build-activation-and-hand-test-boundary
:END:
Keep these gates distinct for every implementation slice:

1. *Source validation*: run QML, shell, JSON, and Python checks against the exact
   source path that would become live.  This is the only gate that sees the
   current out-of-store QML bytes.
2. *Build verification*: for Nix/service changes, evaluate and run
   =nix build --impure --no-link --no-write-lock-file= for the candidate toplevel.
   A successful build is not activation, and in current live-link mode it still
   does not validate QML content.
3. *Explicit activation*: only after review, apply the Nix/Home Manager change or
   deliberately restart the user service.  Record the new generation and PID.
4. *Runtime verification*: inspect =Configuration Loaded=, new warnings, layer
   geometry, restart count, resource delta, and interaction behavior.
5. *Human hand test*: launcher search/launch, clipboard search/delete/preview,
   tray menus, wallpaper selection, media controls, OSD repeat, theme toggle,
   output hotplug, and per-output workspace correctness.

The same tracked checkout already passed the no-link system build recorded in
the parent system audit and evaluates to the active system output.  This focused
audit performed no additional build because it made no Nix or QML candidate
change.  No activation or live service restart was performed.

* Recommended First Implementation Branch
:PROPERTIES:
:CUSTOM_ID: recommended-first-implementation-branch
:END:
Start with Slice 0 and Slice 1 only.  They provide the validation harness, close
the shell-injection boundary, stop the proven temp leak, and fix deterministic
layout/event bugs without changing the approved design.  Re-measure before
centralizing state so every later optimization has a trustworthy comparison.

Do not combine the frame experiment, native status migration, and wallpaper
picker rewrite in one change.  Each touches a different failure domain and has a
different visual/interaction acceptance test.
