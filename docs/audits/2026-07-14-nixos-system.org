* Purpose
:PROPERTIES:
:CUSTOM_ID: purpose
:END:
This is the evidence-backed baseline for the MSI laptop currently reachable at
=192.168.1.175=.  It audits both the live machine and the dotfiles checkout that
produces it, without treating a successful build as proof that a change was
activated.

The audit began read-only.  It did not update the flake lock, edit configuration,
restart services, run =nixos-rebuild test/switch/boot=, change the system profile,
or reboot the machine.  A no-link Nix build realized an already-present output in
the Nix store; that result is recorded separately from active-state evidence.

Evidence labels used below:

- =CONFIG=: declared in the tracked repository or the imported hardware file.
- =BUILD=: Nix evaluation or realization result; not proof of activation.
- =ACTIVE=: observed on the running machine.
- =HISTORY=: journal/profile evidence from an earlier boot or shutdown.
- =INFERENCE=: a conclusion from the preceding evidence, called out explicitly.

** High - network, physical, and boot access
:PROPERTIES:
:CUSTOM_ID: high-network-physical-and-boot-access
:END:
*** Ollama is exposed on every interface
=CONFIG=: Ollama binds =0.0.0.0:11434= and the firewall opens TCP 11434.

=ACTIVE=: =ss= shows =*:11434= and an IPv4 LAN request returned Ollama version
=0.31.1=.  The laptop also has a global IPv6 address and the listener is
dual-stack, but upstream IPv6 reachability was not tested.

=CONFIG/EXECUTED=: the generated firewall start script contains IPv4 and IPv6
accept rules for TCP 11434 and its systemd unit completed successfully.  The
live nftables/iptables ruleset could not be dumped without privilege, so the
rule text is not mislabeled as an independently inspected active ruleset.

There is no authenticated proxy, TLS boundary, or source-subnet restriction in
the repo.  Ollama's local API does not require authentication.  Default to
loopback and close the firewall port unless LAN service is an explicit decision.
If LAN access is required, put it behind a scoped firewall plus an authenticated
proxy or VPN.  See the upstream [[https://docs.ollama.com/api/authentication][authentication documentation]] and
[[https://github.com/ollama/ollama/blob/main/docs/faq.mdx][FAQ]].

Ollama also starts with =OLLAMA_NO_CLOUD=false= and attempted outbound model
cache/recommendation requests to =ollama.com=.  That proves metadata/recommendation
egress attempts, not prompt upload.  If local-only operation is the intent,
decide explicitly whether to set =OLLAMA_NO_CLOUD=true=.

*** The physical session is effectively open
=CONFIG=: TTY1 silently auto-logs into the wheel user and zsh immediately starts
Hyprland.  No idle/lock daemon or locker is declared.  GPG agent credentials may
remain cached for up to eight hours.

=ACTIVE=: no Hyprlock, swaylock, gtklock, Hypridle, or equivalent process/unit is
present; =LockedHint=no=.  Root is a direct ext4 partition with no dm-crypt/LUKS
layer.

Secure Boot protects boot integrity, not stored-data confidentiality or an
already unlocked session.  Add and test lock-on-idle, pre-suspend lock, and
resume behavior; then decide whether autologin still matches the threat model.
Disk encryption is a separate invasive migration/reinstall decision.

*** The ESP random seed is world-readable
=CONFIG/ACTIVE=: =/boot= uses =fmask=0022,dmask=0022=, so
=/boot/loader/random-seed= and its directories are mode 0755.  =bootctl= has
reported this as a security hole on nine observed boots since July 1.

Use a restrictive ESP mount mask such as =umask=0077= (or equivalent file and
directory masks), then separately verify boot entry updates, Secure Boot, and a
real reboot.

** High - confirmed functional and reliability bugs
:PROPERTIES:
:CUSTOM_ID: high-confirmed-functional-and-reliability-bugs
:END:
*** The document portal fails every login
=ACTIVE/HISTORY=: =xdg-document-portal.service= exits with status
=6/NOTCONFIGURED= because neither =fusermount= nor =fusermount3= is on its
execution path.  It cannot mount =/run/user/1000/doc=.  This has recurred across
nine boots since July 2.

The main Hyprland and GTK portals run, but sandbox/document sharing and portal
file integration are incomplete.  Wire =fuse3/fusermount3= through the correct
NixOS/user-unit path, then verify the unit and mount point after activation.

*** Emacs can add 90 seconds to shutdown
=HISTORY=: on the July 14 reboot, =emacs.service= ignored SIGTERM for its default
90-second =TimeoutStopSec=, was SIGKILLed, and left the user session waiting.
Earlier shutdowns also recorded status 15; fourteen Emacs failed-result events
exist since July 1.

=CONFIG=: the unit has =Restart=on-failure= but no =ExecStop= or bounded custom
shutdown policy.

Design a deterministic Emacs shutdown that accounts for unsaved buffers,
processes, and save hooks before shortening the timeout.  A build cannot prove
this fix; it requires a separately planned stop/reboot test.

*** The configured battery governor cannot be selected
=CONFIG=: TLP requests =schedutil= and EPP =power= on battery.

=ACTIVE=: the driver is =amd-pstate-epp=, its available governors are only
=performance powersave=, and its available EPP list currently contains only
=performance=.  While unplugged and discharging, every CPU policy remained
=performance/performance= even though TLP reported successful application.

=schedutil= is invalid for the active driver.  Change only the battery governor
to an available value first, then investigate why the EPP list is restricted
(firmware, ACPI quirks, latest-kernel behavior, or another owner).  Measure
frequency, heat, responsiveness, and battery drain after a separate activation.

** Medium - reproducibility and validation
:PROPERTIES:
:CUSTOM_ID: medium-reproducibility-and-validation
:END:
*** MSI hardware facts are outside the repository
The absolute =/etc/nixos/hardware-configuration.nix= import forces =--impure=,
prevents pure review/builds, and leaves disk/initrd facts outside the checkout.
Copy the verified file to =hosts/msi/hardware-configuration.nix=, review it,
switch to a relative import, and prove that the output path is unchanged before
any activation.

*** Whole-flake checks are red while the invalid host is exported
=desktop-amd= is exported even though its tracked hardware placeholder asserts
false and lacks a root filesystem.  A custom =checks= set cannot make standard
=nix flake check= ignore an invalid =nixosConfigurations.desktop-amd=.  Keep a
deployment guard, but stop exporting the unfinished host as a normal
configuration or redesign its bootstrap representation until it has real
hardware facts.

*** QA ownership is undefined
No formatter, checks, or CI are exported.  Two installed formatters disagree and
currently flag different file sets.  Choose one formatter, add a target-specific
MSI evaluation/build check, shell/JSON/Lua checks, and a deliberate policy for
the deferred AMD host.

*** Home Manager generations do not own most live configs
The live-link model is an intentional fast-edit workflow, but it weakens rollback,
portability, and generation provenance.  Classify each linked surface as stable
or experimental.  Move settled configs to store-backed sources and retain live
links only where immediate edits are worth the tradeoff.

*** The update stack has several simultaneous moving parts
The active host combines NixOS unstable, upstream Hyprland, latest kernel, latest
proprietary NVIDIA, an out-of-tree OpenRazer patch, runtime Emacs packages from
moving archives, and mutable dotfile links.  A single lock update can break boot,
graphics, peripherals, editor startup, or user services together.

Stage lock updates behind parse/eval/build gates, keep a discoverable rollback
generation, and decide whether =linuxPackages_latest= is necessary.  The current
bootloader timeout of zero makes recovery less discoverable even though older
generations exist.

** Medium - performance and resilience
:PROPERTIES:
:CUSTOM_ID: medium-performance-and-resilience
:END:
*** Boot waits on model provisioning
The measured boot is 39.547 seconds: 8.611 firmware, 0.927 loader, 0.632 kernel,
2.201 initrd, and 27.173 userspace.  Graphical target arrives at 26.678 seconds.

=NetworkManager-wait-online= consumes 16.598 seconds.  The critical chain is:

#+begin_src text
graphical -> multi-user -> ollama-model-loader -> network-online
          -> NetworkManager-wait-online
#+end_src

The model loader itself took only milliseconds once online, but its dependency
makes graphical boot wait.  Decouple model hydration from the boot-critical
target before disabling or tightening wait-online.

*** GPU/module initialization is expensive but succeeds
=systemd-modules-load= takes 7.637 seconds and about 982 MiB peak while loading
Razer, AMDGPU, and NVIDIA modules.  It succeeds, so optimize this only after the
network critical path and only with cold-boot/peripheral/GPU evidence.

*** Ollama is CPU-only despite the RTX 4050
Ollama startup reports =compute=cpu= and =total_vram=0 B=; no Ollama process
appears in =nvidia-smi=.  The service has device allowances but the configured
package/runtime does not discover usable GPU acceleration.  Treat CUDA/Vulkan
acceleration as its own slice after closing the network boundary.

*** There is no swap or zram
The machine has 30 GiB RAM and no current OOM incident, but systemd-oomd warns on
every boot that no swap degrades memory-pressure handling.  Compilers, Electron,
two editor stacks, browsers, and LLMs can create sharp peaks.  Add measured zram
first; real hibernation requires a separate correctly sized disk-swap design.

*** The closure is large
The current system closure is 27.0 GiB; Home Manager accounts for an 18 GiB
closure.  Haskell language tooling alone reaches about 7.5 GiB cumulatively and
NVF about 6.5 GiB.  Emacs installs full C/C++, Rust, Haskell, Python, Node, TeX,
QML, and Nix stacks while NVF duplicates part of that toolchain.

This is mostly update/build/store cost rather than idle runtime cost.  Decide the
active editor/toolchain ownership before removing anything; move occasional
language stacks to project dev shells where that improves the workflow.

*** Desktop performance policy favors effects and the discrete GPU
Prime Sync, NVIDIA persistence, blur, VRR/HDR/10-bit display policy, animated
wallpaper support, three Ghostty shaders, large terminal scrollback, and frequent
Quickshell polling all favor responsiveness/appearance over battery life.

The RTX used about 4 W at idle during one sample and owned Hyprland/Xwayland and
desktop clients even while unplugged.  This may be deliberate for a
desktop-replacement workflow.  If mobile use matters, design a separate
AMD-primary/NVIDIA-offload slice with console rollback because compositor GPU
changes can black-screen.

- Persistent journals use about 1.5-1.6 GiB.  Since July 1 they contain over two

- =coredumpctl= has 1,820 historical crash-event records from May 2 through June

  large historical crash population, but not evidence of a current crash loop.
- The Cliphist wipe helper exists but has no key binding, despite the module's

  binding and decide an expiry/size policy.
- The Quickshell flake input is unused; all actual references use
  =pkgs.quickshell=.  Remove it or wire it intentionally.  NVF also carries a
  second Nixpkgs pin rather than following the primary input.
- The global =permittedInsecurePackages = [ "pnpm-10.29.2" ]= exception is stale.

  current build graph uses pnpm 10.34.4 instead.
- =allowUnfree = true= is broad.  A predicate would document which proprietary
  packages are actually accepted, although NVIDIA/Steam/Brave make some unfree
  policy necessary.
- =~/.openai_key= is sourced into every interactive zsh shell.  Its live mode is
  correctly 0600, but scoped injection would reduce how widely the secret enters
  process environments.
- The user is in the broad =input= group for Kanata.  Keep only if direct raw
  input access is required beyond the service.
- Two SSH public keys are declared.  Confirm both are current and intentional as
  ordinary key inventory hygiene.
- IPv4 and IPv6 redirect acceptance are enabled; disable unless this LAN setup
  needs router redirects.  AppArmor, audit, kernel lockdown, kexec restrictions,
  and narrower user namespaces are optional defense-in-depth slices that require
  compatibility testing, especially with proprietary/out-of-tree modules.

** Application and wiring defects to triage
:PROPERTIES:
:CUSTOM_ID: application-and-wiring-defects-to-triage
:END:
- The boot parameters include =acpi=ht=, =acpi_enforce_resources=lax=,
  =acpi_os=linux=, fixed EFIFB geometry, framebuffer reservation, and heavy log
  suppression.  These are high-risk laptop-specific quirks and plausible
  contributors to limited power-policy behavior.  Remove only one at a time,
  with reboot, suspend, battery, audio, keyboard, and GPU checks.
- MSI-specific audio PCI paths, keyboard device paths, TLP battery names,
  NVIDIA bus IDs, Razer devices, boot geometry, monitor layout, and touchpad name
  live in modules inherited by =desktop-amd=.  Move them behind host imports or
  options before the AMD output becomes real.
- =AQ_DRM_DEVICES=/dev/dri/card0:/dev/dri/card1= assumes unstable enumeration.
  Prefer =/dev/dri/by-path= identifiers.
- Kanata exact device paths can fail when the external keyboard is absent or
  renamed.  Current services are running; test cold boot and hotplug separately.
- The fixed Home Manager =.backup= suffix can collide with an older backup on a
  later activation.
- Theme activation runs after =writeBoundary= rather than =linkGeneration=;
  first activation can skip the not-yet-created live script.  The generator also
  updates many files in place, so a mid-run failure can leave mixed theme state.
- Emacs packages come from moving runtime archives without a lockfile, and
  =emacs-sync= compiles vterm inside mutable cache state.  Nix generation rollback
  cannot reproduce that package/cache state.
- =mpris-proxy= is enabled live but not declared.
- Ghostty retains up to ten million scrollback lines and always enables three
  animated shaders.  Firefox permits a 1 GiB memory cache and disables disk
  cache.  Quickshell launches short-lived status commands every two seconds.
  Tune only after profiling real symptoms.
- aerc references =chafa= and =pdftotext= but neither package is declared, and
  its config has two =[ui]= sections.  Its iCloud submission URL uses implicit
  TLS on port 587; no live mail error was found, but account connectivity was not
  tested.  Validate without exposing credentials before changing it.
- NVIDIA VA-API is asserted through session variables and Firefox preferences,
  but =vainfo= is absent, so hardware decoding was not validated.
- =scripts/weather= is executable, starts with a blank line before its shebang,
  and is not referenced by the active configuration.

** Separately authorized live phase
:PROPERTIES:
:CUSTOM_ID: separately-authorized-live-phase
:END:
- =nixos-rebuild test= changes the running system temporarily.
- =nixos-rebuild switch= changes the running system and default profile.
- =nixos-rebuild boot= changes the next boot but not the current runtime.
- A reboot proves the boot path only after it actually occurs.

For an approved live test, preserve SSH/console rollback first, activate the
smallest slice, and collect feature-specific evidence plus:

#+begin_src sh
readlink -f /run/current-system
readlink -f /run/booted-system
systemctl --failed
systemctl --user --failed
journalctl -b -p warning
#+end_src

Do not label a built output active until =/run/current-system= points to it.  Do
not label a switched output boot-verified until =/run/booted-system= points to it
after a reboot.
