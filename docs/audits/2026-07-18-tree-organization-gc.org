#+title: NixOS Tree, Configuration, and Maintenance Audit - 2026-07-18

* Purpose And Safety Boundary
:PROPERTIES:
:CUSTOM_ID: purpose-and-safety-boundary
:END:
This is a read-mostly audit of the current MSI NixOS machine, the live checkout
that configures it, and the matching local checkout.  It focuses on:

- repository and module ownership;
- code and package organization;
- the current Quickshell source and service lifecycle;
- closure growth, generations, caches, logs, and a conservative GC policy;
- a slice-based implementation order.

It complements the broader
[[file:2026-07-14-nixos-system.org][2026-07-14 NixOS system audit]] and the measured
[[file:2026-07-14-quickshell-optimization.org][2026-07-14 Quickshell optimization audit]].
The Quickshell source has not changed since those measurements, so this report
does not duplicate the earlier component-by-component analysis.

No flake update, package edit, garbage collection, cache deletion, build,
activation, service restart, compositor restart, profile change, or reboot was
performed.  Evaluation and =nix flake check --no-build= were kept separate from
live-state inspection.

One nominally read-only command had a small side effect.  While printing GC
roots, =nix-store --gc --print-roots= removed one stale indirect-root link and
five stale temporary-root bookkeeping files.  It did not delete a store path,
profile, generation, or cache, and no garbage collection ran.  Further
dead-path/GC queries were stopped rather than risk more implicit housekeeping.
The Nix reclaim estimate is therefore deliberately non-exact.

Evidence labels:

- =SOURCE=: tracked or dirty checkout bytes.
- =EVAL=: Nix evaluation only; not a build and not activation.
- =ACTIVE=: observed on the running machine.
- =HISTORY=: generation, journal, or coredump history.
- =CONFIRMED=: directly demonstrated defect or ownership mismatch.
- =HYPOTHESIS=: an optimization candidate that still needs comparative
  measurement.

* Executive Verdict
:PROPERTIES:
:CUSTOM_ID: executive-verdict
:END:
The repository does not need a wholesale rewrite.  Its strongest internal
trees, particularly =config/quickshell=, already have understandable
responsibilities and no orphaned QML components.  The highest-value polish is at
the boundaries:

1. Reconcile the live, uncommitted QMK/Vial configuration before doing unrelated
   system work.
2. Move MSI-only facts out of shared modules and bring the MSI hardware file
   into Git.
3. Make package ownership explicit: base runtime, per-user application, optional
   feature, or project development shell.
4. Give theme generation and QML tooling one owner each.
5. Separate stable store-backed configuration from an explicit live-development
   profile.
6. Introduce age-based Nix maintenance only after generation/rollback policy is
   written down.

The machine is not under disk pressure: root is 29% used and inode use is about
2%.  GC is maintenance hygiene, not an emergency.  The largest storage choices
are Steam content, shader caches, ISO downloads, Ollama models, and application
caches; they are not interchangeable with Nix garbage.

The live Quickshell process is currently stable and quiet.  A ten-second sample
used about 1.49% of one CPU core for the complete cgroup, with about 348 MiB
current cgroup memory.  The important new Quickshell reliability finding is not
an independent QML crash: one July 17 Hyprland renderer crash removed the
Wayland display, then Quickshell and hyprpolkit repeatedly restarted too early
and produced dependent coredumps.  Their session ordering/backoff needs a
bounded Wayland-ready gate.

** Stable versus live user configuration
:PROPERTIES:
:CUSTOM_ID: stable-versus-live-user-configuration
:END:
=home/modules/links.nix= declares nine healthy out-of-store links into the live
remote checkout:

- aerc;
- frozen/reference Doom;
- active clean Emacs;
- Ghostty shaders;
- Quickshell;
- runtime scripts;
- the Neovim config overlay;
- WezTerm;
- Hyprland Lua.

This is an effective development workflow, and every live link resolved
correctly.  It is also a validation and rollback boundary: changing those source
bytes can change the active desktop without a Nix build, and a Home Manager
generation does not freeze them.

The correct polish is classification, not blanket conversion:

| Class | Recommended ownership |
|-------+-----------------------|
| stable and rollback-sensitive | store-backed =xdg.configFile= or packaged source |
| actively developed | explicit live-development profile using out-of-store links |
| frozen reference | =archive/= or documentation; no live link |
| runtime state/cache | XDG state/cache paths, never Git source |

** P0 - QMK rules are installed but inert
:PROPERTIES:
:CUSTOM_ID: p0-qmk-rules-are-installed-but-inert
:END:
The dirty system package list contains =qmk-udev-rules=, but package presence in
=environment.systemPackages= does not activate its udev integration.

=CONFIRMED=:

- =hardware.keyboard.qmk.enable=false=;
- =services.udev.packages= omits QMK rules;
- =/etc/udev/rules.d/50-qmk.rules= is absent;
- no =plugdev= group exists;
- the package's rule file is merely exposed under
  =/run/current-system/sw/lib/udev/rules.d/=.

The locked NixOS module's intended interface is
=hardware.keyboard.qmk.enable=true=.  That belongs in the MSI/device-specific
boundary, not the shared global package bucket.

The cost is also disproportionate:

| Addition | Incremental realized closure |
|----------+------------------------------|
| Vial relative to generation 853 | 157,087,776 bytes, about 149.8 MiB |
| QMK CLI/toolchains relative to generation 854 | 1,861,400,008 bytes, about 1.73 GiB |
| Rules package itself | 4,824 bytes |
| All three relative to generation 853 | 2,018,492,608 bytes, about 1.88 GiB |

Least-bloated ownership:

- enable QMK device rules through the NixOS option on the MSI host;
- keep Vial as a user application if it is regularly used;
- place the full =qmk= compiler/flasher stack in a project dev shell or invoke
  it on demand unless it truly belongs on every system generation;
- remove the explicit =qmk-udev-rules= package once the module owns it.

Moving Vial from NixOS to Home Manager clarifies ownership but may not reduce the
integrated NixOS closure by itself; measure rather than assume.

** P1 - compositor recovery amplifies into dependent desktop crashes
:PROPERTIES:
:CUSTOM_ID: p1-compositor-recovery-amplifies-into-dependent-desktop-crashes
:END:
The recent coredump population is not nineteen independent current regressions.

On July 17, Hyprland hit one renderer assertion and lost the Wayland display.
The portal then segfaulted; Quickshell and hyprpolkit restarted while the display
still refused connections.  Quickshell produced nine coredumps and hyprpolkit
five before session shutdown/start limits intervened.

On July 18, the second Hyprland coredump occurred during an intentional
compositor/session stop.  Quickshell, polkit, and the portal stopped normally
and restarted successfully.  Brave and Electron =SIGTRAP= dumps are in that same
maintenance window and should not be mislabeled as current independent loops.

=ACTIVE=: on the current boot Quickshell, Hyprland, portal-hyprland, and
hyprpolkit are running with no new restart.  The separate
=xdg-document-portal.service= remains failed for the older missing-fusermount
bug.

The repository-owned Quickshell improvement is a bounded Wayland-ready
=ExecStartPre= or equivalent dependency, plus deliberate restart delay/backoff.
=After=graphical-session.target= alone did not protect an in-place compositor
restart.  Apply the same lifecycle reasoning to hyprpolkit through an override
or upstream/NixOS ownership.  Preserve the recent core files until the primary
Hyprland crash is understood.

** Conservative Nix maintenance policy
:PROPERTIES:
:CUSTOM_ID: conservative-nix-maintenance-policy
:END:
The currently available NixOS options support an age-based policy:

#+begin_src text
nix.gc = {
  automatic = true;
  dates = "weekly";
  options = "--delete-older-than 30d";
  randomizedDelaySec = "45min";
};

nix.optimise = {
  automatic = true;
  dates = "weekly";
};

boot.loader.systemd-boot.configurationLimit = 10;
#+end_src

This is a proposal, not an applied patch.  Treat the three concerns separately:

1. *Generation policy*: retain a known-good boot generation and document why
   each exceptional generation is protected.
2. *GC policy*: delete only roots older than the chosen age, then collect
   unreachable paths.
3. *Store optimization*: deduplicate identical files; measure duration and I/O
   independently from GC.

For the first activation, prevent an immediately caught-up maintenance timer or
schedule it explicitly in the future.  Inspect generated units and timers before
the first run.  The first actual collection is a separately authorized,
observed maintenance event with before/after store, free-space, generation, and
boot-entry evidence.

* Verification Contract
:PROPERTIES:
:CUSTOM_ID: verification-contract
:END:
Every implementation slice should record five distinct gates:

1. *Source validation*: exact dirty diff, parsers/linters, reference checks, and
   generated-output comparison.
2. *Evaluation*: target-specific =nix eval=; not a build.
3. *Build verification*: no-link build and closure comparison; not activation.
4. *Live activation*: separately authorized =test=, =switch=, =boot=, or user
   service restart.
5. *Runtime and hand tests*: active store path, units, logs, hardware/app
   behavior, and rollback.

A green build never proves that the machine is running it.  An out-of-store
Quickshell/Emacs/Hyprland link also means a green Nix build does not prove the
active source bytes unless that slice adds its own source validation.
